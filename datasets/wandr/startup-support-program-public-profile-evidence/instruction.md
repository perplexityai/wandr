## Universal rules

The following rules apply to every task below.

**Identifier discipline.** Same entity → same string. Different entities → different strings. When you're unsure whether two names refer to the same thing (spelling variants, model editions, product versions), don't offload the ambiguity to the user — decide and commit. Don't hedge by splitting "just in case" or merging "probably close enough." Both failures cost credit.

**More is (usually) better.** Whenever the task says "at least N" / "N+" / etc, going past N generally helps your score — treat those as soft floors, not exact targets.

**No duplicate entities.** Do not, however increase volume via duplicate entities, all the entities must be meaningfully different, entity-duplicating rows will generally be penalized; in particular, do not supply multiple rows per entity to "supply the answer in chunks", which will also be treated as entity duplication.

**Every `url` you submit must be fetchable.** Do not submit URLs you expect to be non-resolvable (DNS failure, dead host) as your `url`. Evidence should come from an available `url` (even if you wished to, say, provide evidence for some URL's unhealthiness).

**Every row carries `excerpts`** — verbatim or near-verbatim quotes from the source page (whitespace, punctuation, ellipses to skip irrelevant clauses are fine) **with semantics preserved**. An excerpt is what the page literally says, in the meaning the page intends. Fabrication, paraphrase that shifts meaning, sentence-stitching across sections, or selective cropping that flips a hedge into confidence — all fail.

The excerpts collectively make the answer evident. *Every* task-required claim / task-asked question / answer field / etc MUST have its support visible somewhere in the excerpt set — not just nearby on the page. The reader's test: imagine someone sees only your excerpts (with no access to the rest of the page); can they verify each piece of your answer? If a page genuinely doesn't carry what the task asks for, find a different page or skip the entity rather than fish for tangential excerpts. If you deem paraphrasing necessary / desirable for proper answer delivery, that's admirable and encouraged: paraphrase to your heart's desire within `answer` fields, make new `answer` fields and redistribute summaries among them as you see fit, but excerpts stay faithful and fully evidence-complete.

**Page contents only.** This is a task about citing web pages for human consumers, and citations are expected to be human-usable — both in where they are sourced from and in how well they stand on their own, out of page context. Excerpts come from the web-page main text — what a human reader sees on the page. Excerpts should also look sensible by themselves, with their information-bearing intent clear. API response blobs, page metadata fields (timestamps, view counts, score numbers), structured-data payloads (`__NEXT_DATA__`, JSON-LD, OpenGraph), and other “robot-side” sources / page representations are out of scope. In a similar vein, be wary of citing image captions / on-hover alt text / infoboxes / specially rendered bibliography or reference units / UI or navigation elements / etc. (unless confident in both their visibility and critical utility for the task), and avoid citing image contents, hyperlink-encoded URLs, and similar evidence surfaces altogether: anything outside the straightforward “main body of text” risks reducing citation ergonomics to the point where it is considered unusable.

**Signaling absence.** If you mean for a blank or sentinel `answer` field to assert "this required information isn't on the page" (vs. "I missed it"): first verify the task warrants such an option — many tasks treat blank-required as an invalid entity. When absence IS admitted, flag the intent explicitly in an appropriately-named `answer` field, and let your excerpts carry the strongest available evidence — direct proof-of-absence ("not listed", "n/a") if the page provides it; otherwise, try at least capturing the page segments where the missing info would plausibly have appeared if it existed, where applicable.

## `startup_support_program_public_profile_evidence`

For 120+ durable public startup-support programs, cover each of the 4 source-evidence roles listed below by supplying a public source (i.e. 1+ URL under each role) that visibly supports that role for the named program.

The aim is public provenance for accelerators, incubators, pre-accelerators, startup centers, venture builders or studios, and comparable university, corporate, government, nonprofit, or ecosystem startup-support programs.

Evidence roles:
- `official_program_surface`: a current program-owned, host-owned, operator-owned, or official institutional page dedicated to the named program and presenting its startup-support role.
- `official_activity_or_terms`: an official or program-authorized page showing a dated 2024-2026 program activity, admission/application cycle, cohort, demo day, public event, program period, or specific participant-facing terms such as concrete benefits, funding, equity, eligibility, target stage, sector focus, duration, deadline, cohort timing, or support offer.
- `sponsor_operator_counterparty`: a page controlled by a named host, sponsor, operator, funder, corporate/government/university partner, admitted startup, alumnus, or comparable counterparty that states a program-specific relationship with the named program.
- `independent_ecosystem_context`: an independent publisher, ecosystem organization, city/economic-development source, association, award body, trade publication, news outlet, report, or comparable non-program-controlled surface whose page is primarily about the named program or a narrow ecosystem story involving it; a multi-program page fits only when it is a narrow ecosystem/report/award/association/city story and the cited section gives non-generic program-specific rationale, role analysis, impact, or contribution.

The named entity ought to be a durable public startup-support program/entity. Pure VC funds, generic coworking spaces, events/conferences, broad communities, coupon/perks pages, generic companies, ranking/listicle/advice pages, and generic application guidance are out of scope unless the cited page visibly establishes a concrete startup-support program role. Cohort, batch, class, application page, challenge round, dated program instance, or intake campaign titles are not standalone root programs unless the cited page establishes that title as a durable startup-support program entity; otherwise that evidence belongs to the parent program's activity or terms. Broad multi-program lists, tables, rankings, encyclopedia pages, and directories are too thin when they only provide table columns, profile fields, name/type labels, founding-year cells, headquarters, generic descriptions, scraped summaries, reusable per-program blurbs, recommendation tiles, or "best for" summaries. They fit an evidence role only when a program-specific section gives role-appropriate relationship, activity, terms, or ecosystem treatment that is substantially richer than catalog/list/profile data.

Requirements:
- The page must clearly identify the named program/entity. Parent institutions, operators, sponsors, portfolio companies, similarly named programs, and one-off cohort/application titles are not enough unless the named program itself is visible as a durable startup-support entity.
- The page should make its `evidence_role` source role visible. For `official_program_surface`, the source role comes from program-owned, host-owned, operator-owned, or official institutional control and a dedicated program context; generic parent navigation, operator-wide directories, archived announcements, and independent lists do not pass. For `official_activity_or_terms`, the page should be official or program-authorized and should visibly carry dated 2024-2026 cycle/activity framing or specific participant-facing terms for the named program; generic official marketing text, undated "apply now" buttons, and vague promises of mentorship/resources are insufficient. For `sponsor_operator_counterparty`, the page should be controlled by, or centrally about, a distinct host/sponsor/operator/funder/partner/admitted-startup/alumnus/counterparty and should state that party's relationship with the program; the program's own generic homepage and generic third-party catalog profiles are insufficient by themselves. For `independent_ecosystem_context`, the page should have independent editorial, ecosystem, institutional, award, association, report, or news context, and it should be primarily about the named program or a narrow ecosystem story involving it. A broad "best accelerator" page, SEO listicle, ranking, shallow city/category list, startup-advice article, hosted profile, encyclopedia entry, or multi-program directory/catalog page is insufficient even when it contains a paragraph-level program profile, pros/cons tile, recommendation note, or "best for" summary. A multi-program page can pass this role only when the page is a narrow ecosystem/report/award/association/city story and the cited section gives non-generic rationale, role analysis, impact, or contribution for the submitted program.
- The page must expose the selected role's public-program evidence for the named program. For `official_program_surface`, it must identify the startup-support role from an official, dedicated program context. For `official_activity_or_terms`, it must show a dated 2024-2026 activity/admission/cohort/application/demo-day/event/program-period signal or specific terms such as benefits, funding, equity, eligibility, target stage, target sector, duration, deadline, cohort timing, or support offer. For `sponsor_operator_counterparty`, it must state a program-specific host, sponsorship, operating, funding, partnership, participation, admission, portfolio, alumni, or comparable counterparty relationship, not merely logos, founder names, mixed backer/founder cells, or unexplained name lists. For `independent_ecosystem_context`, it must place the program in a broader startup ecosystem with program-specific analysis, award rationale, city/industry role, ecosystem contribution, reported impact, or comparable narrative substance beyond name/type/location/profile facts, rankings, recommendation copy, program terms, or generic benefits.
- The page should have enough role-specific source depth for `independent_ecosystem_context` that its cited section would still read as meaningful ecosystem analysis if separated from a surrounding list. A URL designed to list, rank, compare, or catalog many startup programs through parallel entries is invalid for this role when the submitted program is only one reusable catalog/list/ranking entry among many.
